# Ladder Generation & Wall Clearance Plan

## Overview

This document details the current logic for each ladder type in the connectivity pipeline, the existing wall deletion mechanism used for forced connections, and a proposed extension of that mechanism to clear blocking walls at the top of all ladder types.

---

## Part 1 — Ladder Types

There are five ladder types. Four are generated by dedicated functions; tower ladders are a special case of ground ladders placed in a final pass.

---

### 1. Yellow Ladders (`type: 'ladder'`)

**File:** `src/generators/connectivity/generate-yellow-ladders.js`  
**Helpers:** `src/generators/connectivity/generate-yellow-ladders-helpers.js`  
**Debug colour:** Yellow (`#cccc22`)

**Trigger:** A culled walkway has `blocked: true` — meaning its path is obstructed by walls. Yellow ladders provide the vertical leg that allows movement to continue over the obstacle.

**Placement logic (`buildYellowLadderAtEndpoint`):**

1. Inspect both `start` and `end` endpoints of each blocked walkway.
2. Cast a small probe rect (0.3" margin) at the endpoint along the walkway axis.
3. Walk `data.walls` to check whether any wall at `tier * tierHeight + slabThickness` touches the probe. If no wall is found, abort — the endpoint does not need a ladder.
4. Position the ladder body pressed against the wall with a `wallOffset` (0.3") gap:
   - X-axis walkway: ladder width = `LADDER_DEPTH`, depth = `LADDER_WIDTH`; positioned at the blocked end
   - Z-axis walkway: ladder width = `LADDER_WIDTH`, depth = `LADDER_DEPTH`
5. Find the highest tier where a wall exists at the ladder footprint (`findHighestWalledTier`).
6. Walk downward from `topTier + 1` until a floor section overlaps the ladder footprint — that becomes `ladderTopTier`.
7. `y0 = walkway.y`, `y1 = ladderTopTier * tierHeight`. Abort if `y1 <= y0`.
8. `platformDir` is the direction away from the wall and toward the walkway.

**Key constraints:**
- Discarded if the parent walkway is later deleted.
- No random culling ratio applied (unlike red/orange/cyan).

### 2. Ground / Red Ladders (`type: 'ground_ladder'`)

**File:** `src/generators/connectivity/generate-ground-ladders.js`  
**Helpers:** `src/generators/connectivity/generate-ground-ladders-helpers.js`  
**Debug colour:** Red (`#cc4444`)

**Trigger:** Every external quadrant edge of every non-tower building at ground level (tier 0). "External" means the neighbouring quadrant in that direction either does not exist or is absent from the tier-1 quadrant set.

**Placement logic (`buildGroundLadderForEdge`):**

1. Centre the ladder on the quadrant edge (`edge.len / 2 - LADDER_WIDTH / 2`), offset outward from the wall by `wallOffset` (0.3").
2. Check `data.walls` at `slabThickness` (ground-tier wall Y). If no wall exists at the footprint, return `null`.
3. Walk tiers upward from 1 to `config.tiers`. At each tier, check whether a wall exists at `t * tierHeight + slabThickness`. Stop ascending on the first tier with no wall. Record the last tier that had a wall as `topTier`.
4. Walk downward from `topTier + 1` until a floor section overlaps the footprint. That is `ladderTopTier`.
5. `y0 = 0`, `y1 = ladderTopTier * tierHeight`. Abort if `y1 <= 0`.

**Post-generation filter:**
- If `DELETIONS.redLadderWalkwayOverlap` is enabled, any ground ladder whose footprint intersects a culled walkway is discarded.
- Culled to 60% of candidates (`CONNECTIVITY.ladderCullRatio`) in a later shared cull pass.
- Proximity cull removes red ladders too close to orange ladders.

---

### 3. Orange Ladders (`type: 'orange_ladder'`)

**File:** `src/generators/connectivity/generate-orange-ladders.js`  
**Debug colour:** Orange (`#ee8822`)

**Trigger:** Randomly spawned on external quadrant edges at any tier (0–`config.tiers - 1`), with tier-dependent spawn chance:
- Tier 0 (ground): 10% (`CONNECTIVITY.orangeSpawnChance.ground`)
- Tier 1: 20% (`CONNECTIVITY.orangeSpawnChance.tier1`)
- Tier 2+: 30% (`CONNECTIVITY.orangeSpawnChance.tier2Plus`)

**Placement logic:**

1. Identical edge enumeration to ground ladders — external quadrant edges, respecting `mapBoundaryMargin`.
2. Roll the RNG against the tier-appropriate spawn chance; skip on failure.
3. Position ladder body at the same `wallOffset` formula as ground ladders.
4. Compute `y0 = tier * tierHeight`.
5. Walk upward from `tier + 1`, checking whether a floor section overlaps the footprint at each tier. Stop on the first tier with no matching floor. Record the last tier that had a floor as `topTier`.
6. Require `topTier - tier >= CONNECTIVITY.orangeMinSpan` (default 2); discard otherwise.
7. `y1 = topTier * tierHeight`.

**Post-generation filter:**
- Culled to 60% of candidates.
- Discarded if footprint touches a walkway or a red ladder.
- Discarded if top is within `ladderTopWalkwayDist` (2") of a walkway.
- Labelled `'orange_ladder_BAD_'` in debug if no floor exists at the termination tier (should not survive the span check but is a safety marker).

### 4. Interior / Cyan Ladders (`type: 'interior_ladder'`)

**File:** `src/generators/connectivity/generate-interior-ladders.js`  
**Helpers:** `src/generators/connectivity/position-interior-ladder.js`, `src/generators/connectivity/find-interior-ladder-tier-range.js`  
**Debug colour:** Cyan (`#22cccc`)

**Trigger:** Interior quadrant edges at tier 2 and above, where a neighbouring quadrant within the same building is absent at that tier. These serve movement across floor-level step changes inside buildings.

**Placement logic:**

1. Iterate over all buildings, tiers 2–`maxTier`, present quadrants.
2. For each of the four neighbour directions, if the neighbour quadrant index is valid but absent from `bq.tiers[tier]`, this is a candidate edge.
3. `findInteriorLadderTierRange` determines `baseTier` and `topTier` by looking at what tiers the source and missing quadrants are present across.
4. `positionInteriorLadder(qr, side)` computes `{x, z, w, d}` with dimensions rotated: for north/south edges the ladder is `LADDER_WIDTH × LADDER_DEPTH`; for east/west it is `LADDER_DEPTH × LADDER_WIDTH` (transposed from external ladders). The body extends toward the missing quadrant.
5. Walk downward from `topTier` until a floor section overlaps the footprint — that is `trimmedTop`.
6. Discard if `trimmedTop <= baseTier`.
7. `y0 = baseTier * tierHeight`, `y1 = trimmedTop * tierHeight`.

**Post-generation filter:**
- Culled to 40% of candidates (`CONNECTIVITY.cyanLadderCullRatio`).
- Discarded if footprint overlaps an orange ladder.
- Discarded if top is within `ladderTopWalkwayDist` (2") of a walkway.

---

### 5. Tower Ladders

**File:** `src/generators/connectivity/generate-tower-ladders.js`  
**Debug colour:** Red (same as `ground_ladder`)

**Trigger:** Any building with `size === 'tower'`. Generated in a final pass after all other ladder types so they can avoid conflicts with existing ladders.

**Placement logic:**

1. Collect all existing ladders from every previous category into `allExistingLadders`.
2. For each tower, compute `topTier = building.maxTier` (or `maxTier - 1` if `pyramidRoof`).
3. Shuffle the four cardinal sides; try each in order.
4. For each side, position the ladder body at the building exterior edge (centred, `LADDER_WIDTH × LADDER_DEPTH`).
5. Check map bounds; skip if out of range.
6. Check AABB overlap (0.3" margin) against all existing ladders; skip if overlapping.
7. Place the ladder: `{ type: 'ground_ladder', y0: 0, y1: topTier * tierHeight }`.
8. **Immediately delete** any wall segments whose `baseY >= exitY` and whose AABB overlaps the ladder footprint (with 0.3" margin). This is an inline reverse-splice loop over `data.walls`.
9. Break after placing one ladder per tower.

The wall deletion at step 8 is the only place where ladder placement directly modifies `data.walls` today (outside the proposed extension below).

## Part 2 — Forced-Connection Wall Deletion

**File:** `src/generators/gap-detection/clearBlockingWalls.js`  
**Called by:** `src/generators/gap-detection/tryForceConnection.js`

When a forced walkway (gap-bridging connection) is created between two buildings, the function `clearBlockingWalls` is called at **both endpoints** of the walkway to remove wall segments that would physically block passage through the wall face.

### Detection algorithm

```
clearBlockingWalls(candidate, endpointX, endpointZ, facingDir, state)
```

1. Determine the walkway's **cross-axis span** (`crossMin`/`crossMax`) — the width perpendicular to travel direction.
2. Scan every wall in `state.walls`:
   - **Tier filter:** `wall.baseY` must fall within `[tierY - tierH + 0.1, tierY + 0.1]`.
   - **Axis filter:** East/west faces require a Z-axis wall; north/south faces require an X-axis wall.
   - **Position filter:** The wall's near or far X (or Z) edge must be flush with the endpoint coordinate (within `wallThickness + 0.5"`).
   - **Overlap check:** Compute the 1-D overlap of the wall's length against `crossMin`–`crossMax`. If overlap > 0, record `{ index, overlap }`.
3. Sum all `overlap` values. Divide by `walkwaySpan` to get `coverage`.
4. If `coverage > 0.5`: delete every recorded wall segment by index (sorted descending for safe `splice`).
5. If `coverage <= 0.5`: leave all walls — they don't meaningfully block passage.

### Key properties
- **Threshold-gated:** Only deletes when the blockage is substantial (>50% of the walkway width). Minor overlaps are left alone.
- **Both ends:** Called twice per forced walkway — once at the source building face, once at the destination.
- **Non-destructive on partial overlap:** If a wall only clips the walkway edge it is preserved.

## Part 3 — Proposed: Top-of-Ladder Wall Clearance

### Problem

All non-tower ladders climb to a termination floor. At the top, a wall segment often sits flush against the building edge at exactly the tier where the ladder exits. This wall blocks a player stepping off the ladder onto the floor above — the same problem that `clearBlockingWalls` solves for walkway endpoints.

Currently only tower ladders delete this wall (inline in `generate-tower-ladders.js`). All other types leave it intact.

### Proposed mechanism

Introduce a shared helper function — analogous to `clearBlockingWalls` — that removes wall segments at the top exit point of any ladder:

```
clearLadderTopWall(ladder, state)
// ladder: { x, z, w, d, y1, platformDir }
// state: { walls, config }
```

**Algorithm:**

1. The exit tier is `round(ladder.y1 / tierHeight)`. The exit wall Y is `exitTier * tierHeight + slabThickness`.
2. Determine the **face direction** the ladder approaches from — this is `ladder.platformDir` (already stored on every ladder type).
3. From `platformDir`, derive:
   - The **axis** of the wall to look for (perpendicular to travel direction).
   - The **coordinate** the wall must be flush with (the building edge the ladder exits through).
4. Scan `data.walls`:
   - **Y filter:** `wall.baseY` within 0.5" of `exitWallY`.
   - **Axis filter:** Same axis-from-direction logic as `clearBlockingWalls`.
   - **Position filter:** Wall edge flush with the exit face coordinate (within `wallThickness + 0.5"`).
   - **Overlap filter:** 1-D overlap of wall length vs. ladder cross-axis span.
5. If total overlap / ladder cross-span > 0.5: delete all overlapping wall segments (reverse splice).

This mirrors `clearBlockingWalls` exactly, with the walkway replaced by the ladder footprint and the tier height replaced by `y1`.

### Call site

After each ladder type is finalised and before cull passes (or alternatively after cull, to avoid deleting walls for ladders that are subsequently removed), call `clearLadderTopWall` for every surviving ladder. The natural location is the end of the connectivity generation pass, in the same block where ladder platforms are appended.

A config toggle should be added to `DELETIONS` so the behaviour can be disabled for debugging:

```js
// src/config.js DELETIONS section
ladderTopWallClearance: true,
```

## Part 4 — Impact on Existing Ladder Logic

### Yellow ladders

Yellow ladders already require a wall to exist at the endpoint (`touchesWall` check). The top-exit wall is at a different Y (`y1`) than the walkway wall (`y0 + slabThickness`), so the detection probe targets a different tier. **No conflict.** The wall the yellow ladder validates against is at the bottom; the wall to be cleared is at the top.

### Ground / Red ladders

Ground ladders use `buildGroundLadderForEdge`, which verifies walls at `slabThickness` (ground tier) and each ascending tier to set `topTier`. The wall at `topTier * tierHeight + slabThickness` is exactly the one blocking exit. Clearing it does not affect the tier-walk logic because that logic runs first. **No conflict.**

### Orange ladders

Orange ladders use a different top-detection strategy: they walk upward through floors rather than walls. The top wall is incidental — it happens to exist at the building face but is not used by the generation logic. Clearing it post-generation is safe. **No conflict.**

However, orange ladders are subject to post-generation culling. Wall clearance should happen **after** the cull pass to avoid deleting walls for ladders that are ultimately removed.

### Cyan / Interior ladders

Interior ladders exit at internal quadrant edges. Their top wall is an inward-facing wall on a floor step rather than an external building face. The `platformDir` stored on interior ladders points toward the missing quadrant, which correctly identifies which internal face wall to clear. **No conflict,** but edge-case testing is warranted for the internal face coordinate derivation.

### Tower ladders

Tower ladders already perform their own inline wall deletion at `y1`. If `clearLadderTopWall` is applied universally, tower ladders would run the clearance twice. The second pass would find nothing to delete (already cleared) and exit cleanly. **No conflict,** but the inline deletion in `generate-tower-ladders.js` could be removed in favour of the shared helper for consistency.

### Cull ordering dependency

The most important ordering constraint is:

```
generate all ladders → run all cull passes → clearLadderTopWall for each survivor
```

Clearing walls for ladders that are subsequently culled leaves holes in walls with no corresponding ladder. This is cosmetically wrong and potentially breaks sightline guarantees. The call must happen after final survivor sets are established.
